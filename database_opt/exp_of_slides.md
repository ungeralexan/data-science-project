# How would we store the data
The idea is to use Supabase; which is basically a hosted PostgreSQL database that comes with built-in authentification, permission and automatically generated API.
So instead of setting up our own backend server, Supabase gives us both the database and a ready to use REST API to insert query or update the data, directly from python for instance. 
It moreover also enforces rowlevel security which means every student only sees their own data.
Thats impottant because some of the data originally comes from university emails, so we want to make sure that nothing leakes between users.

- It is central based:
All events, profiles, and interests live in one consistent schema, which means we can easiliy filter and join the data.

- Autogenerated rest api
Supabase turns every table into an endpoint automatically. We can use it directly from Python or our frontend.

- Row level security
This ensures students can only query their own records, which is critical or privacy and GDPR compliance.

- Several tables
We’ll likely have tables for students, events, interests, and matching results. That keeps things modular and clean.

# Why supabase
We chose Supabase because it’s SQL-based, so it fits well with what we already know, and it adds useful extras like user authentication and a simple web interface to monitor data. 

- It is SQL based:
I can query and join data directly in SQL, whihc is perfect for analytics and debugging

- Integrated authentifications and permissions
Built-in login system that connects with RLS, so we don’t have to write our own user management.

- Web UI
The web dashboard let us quickly view tables, test queries and adjust data without needing code

# What about the capacity
The free tier of Supabase is more than enough for our prototype. We get 500 MB of database storage, which can hold hundreds of thousands of events, and 1 GB of file storage for things like event images. It is ideal for testing and we dont expect to hit these free limits soon.

# DIfference betwen PostgreSQL and Supabase
PostgreSQL is the database engine — like the raw machine that stores and manages data.
→ You can think of it as the “engine under the hood”.

Supabase is a platform built around PostgreSQL that adds extra tools on top:
- web dashboard (to view and manage tables easily)
- Automatic API (so your app or Python script can talk to the database directly — no backend code needed)
- Authentication system (login, users, permissions)
- Row-Level Security (RLS) (so users only see their own rows)
- Hosting + backups (they run the server for you, you don’t have to install anything)


# Row level security
We (the project team) have admin access.
- We can still see everything in the database — all users, all events, all matches.
- This is useful for debugging and managing the data.

The students and users that log in into the account
- Because of RLS, the database itself automatically filters what they can see.
- So if a student sends a query like “get my events,” Supabase will internally check their user ID and only return the rows that belong to them.

# The broad picture a bit 
Your dashboard (web app) is build probabaly with streamlit hence has a login screen (email + password, or university email)
This is the only place students the can interact with
When a student logs in the dashboard sends their login info to Supabase Auth (via the Supabase library or API).
Supabase then checks the credentials and if correct, it sends back a session token (like an ID card saying “this is student X”).
Every time the dashboard requests data (for example: “show me upcoming events”), it sends that token to Supabase.
Supabase then verifies the token and applies Row level security, which mean it could automatically filter rows for the specific user
